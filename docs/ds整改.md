# DolphinScheduler 整改设计方案

> 文档版本：v1.0 | 日期：2026-05-11
> 定位：产品设计 + 技术落地方案

---

## 一、现状问题

### 1.1 体验割裂

当前「调度中心」通过 iframe 嵌入 DolphinScheduler 原生 UI。用户从 Portal 进入后：
- 看到 DS 自己的登录页（已通过自动登录解决）
- 进入后出现 **双层导航**（Portal 侧边栏 + DS 自带顶栏/侧栏）
- UI 风格完全不同（DS 是深蓝色 + Element UI，Portal 是浅白色 + Arco Design）
- 操作逻辑断裂：在 Portal 里点来点去突然进入另一个系统

### 1.2 功能冗余

DS 作为一个完整的调度平台，自带大量本项目不需要的功能：

| DS 功能 | 是否需要 | 处理方式 |
|---------|---------|---------|
| 工作流编排（DAG 编辑器） | 需要 | 保留，通过模态框内嵌 |
| 工作流执行/调度 | 需要 | 拆出来做原生页面 |
| 任务日志查看 | 需要 | 拆出来集成到调度中心 |
| 资源中心（文件管理） | 需要 | 后续做「数据开发」页面时集成 |
| 数据源管理 | 重复 | 与 Portal 数据源合并，Portal 为主 |
| 告警通知 | 需要 | 通过 Webhook 集成到 Portal 通知 |
| 用户管理 | 不需要 | Portal 已有用户体系，去掉 |
| 租户管理 | 不需要 | 单租户场景，去掉 |
| 项目管理 | 不需要 | 后端固定一个项目，去掉 |
| 环境管理 | 不需要 | 不涉及多环境，去掉 |
| 界面设置/时区/语言 | 不需要 | 去掉 |
| Worker 分组 | 不需要 | standalone 模式用不到，去掉 |

### 1.3 信息断层

- 工作台看不到昨天调度跑了多少、成功多少、失败多少
- 调度失败了要进 DS 后台才能看日志、重跑
- 系统监控里 DS 的状态是硬编码的假数据

---

## 二、改造目标

**一句话：把 DS 的壳去掉，把有用的肉拆出来，融进 Portal 的骨架里。**

具体目标：
1. 用户打开 Portal 就能看到调度执行全貌，不需要进第二个系统
2. 调度操作（查看、运行、停止、重跑、看日志）全部在 Portal 内完成
3. 只有「创建/编辑工作流」这种复杂操作才弹出 DS 编辑器
4. 所有页面风格统一，感觉是一个产品而不是拼凑
5. 去掉所有不需要的 DS 功能，减少认知负担

## 三、用户旅程设计

### 3.1 典型角色

**数据工程师**：日常工作是建数据管道、写 ETL 脚本、排查调度问题。
- 主要用到：DataX 同步任务、SQL/Python/Shell 脚本、CRON 调度
- 不关心：多租户、Worker 分组、复杂告警策略

### 3.2 核心场景

#### 场景 A：早晨巡检（最高频）

```
打开 Portal
  → 工作台：一眼看到昨日调度概览
    ├─ 45 次执行 | 40 成功 | 3 失败 | 2 未运行
    └─ 圆环图直观展示比例
  → 看到有 3 个失败（红色醒目）
  → 点击「3 失败」或直接进「调度中心」
  → 工作流列表，失败的排在最前面（红色标记）
  → 点击失败的工作流「查看历史」
  → 抽屉展开，看到最近运行记录
  → 点击失败的那次运行
  → 看到里面 5 个任务，第 3 个失败了
  → 点击任务查看日志 → 等宽字体显示完整日志
  → 定位问题后点击「重跑」
  → 确认 → 工作流重新执行
  → 完成，关闭抽屉，回到列表
```

**关键设计原则：** 从发现问题到解决问题，不超过 5 次点击，全程不离开 Portal。

#### 场景 B：新建调度任务

```
调度中心 → 点击「新建工作流」
  → 弹出全屏模态框（内嵌 DS DAG 编辑器）
  → 在 DS 编辑器里拖拽节点、配置任务（DataX / SQL / Shell）
  → 保存
  → 关闭模态框 → 列表自动刷新，新工作流出现
  → 点击「上线」→ 设置 CRON → 开始调度
```

**设计取舍：** DAG 编辑器是 DS 最核心最复杂的功能，重写代价极高且没必要。用模态框 iframe 内嵌是性价比最高的方案——用户只在创建/编辑工作流这个低频操作时接触 DS 原生界面，其他高频操作全部原生化。

#### 场景 C：系统巡检

```
系统监控 → 看到所有服务状态（真实健康检查）
  → DS 区域显示 CPU 45%、内存 68%（进度条+颜色）
  → 一切正常，绿色
```

## 四、信息架构调整

### 4.1 导航结构

```
├── 工作台              ← 增加调度执行统计（Phase 1）
├── 数据集成
│   ├── 数据源管理      ← 后续与 DS 数据源合并（Phase 5）
│   └── 同步任务
├── 调度中心            ← 改为下拉子菜单（Phase 2）
│   ├── 工作流管理      ← 配置态：创建、编辑、上下线、CRON
│   └── 运行记录        ← 运维态：执行结果、日志、重跑、补数
├── 数据治理
│   ├── 数据资产
│   └── 数据血缘
└── 系统监控            ← 增加 DS 资源监控（Phase 4）
```

**调度中心拆成两个子页面的理由：**

| | 工作流管理 | 运行记录 |
|--|----------|---------|
| 使用频率 | 低频（一周几次） | 高频（每天看） |
| 用户心智 | "我要配置/上线一个任务" | "昨天跑得怎么样" |
| 对应 DS 页面 | Process Definitions | Process Instances |
| 同行参考 | DataWorks 调度中心 | DataWorks 运维中心 |

这也让菜单结构更均衡——数据集成、调度中心、数据治理都是下拉，视觉上一致。

运行记录单独成页的最大好处：早晨巡检时，用户看的是**所有工作流的执行情况**，不需要逐个点开。运行记录页直接展示全量执行记录，筛选失败，一目了然。

### 4.2 通知系统（新增能力，不新增入口）

顶部导航栏的通知铃铛从装饰变成实际功能：
- DS 工作流失败 → 自动推送通知到铃铛
- 红点提示未读数量
- 点击展开通知列表，可一键跳转到对应工作流

通知不单独做页面，融入已有的交互入口（铃铛），保持界面简洁。

## 五、各页面改造详细设计

### 5.1 工作台（Dashboard）— Phase 1

#### 改造前

4 张统计卡片（数据源数量、同步任务数量、调度工作流、数据资产），数字来自 Portal 数据库，和 DS 没有任何关系。下面是快捷入口、公告、架构图。

#### 改造后

**统计卡片（4 张，重新定义含义）：**

| 卡片 | 主数字 | 副信息 | 迷你图 |
|------|--------|--------|--------|
| 数据源 | 总数 | X 个可用 | 不变 |
| 同步任务 | 总数 | X 个运行中 | 不变 |
| 调度工作流 | 工作流总数 | 已上线 X 个 | 近 7 天每日执行数趋势线 |
| 昨日执行 | 昨日总次数 | 成功 X / 失败 X | 近 7 天成功率趋势线 |

**新增：昨日调度概览区域**

紧接统计卡片下方，一张 glass-card：

```
┌─────────────────────────────────────────────┐
│  昨日调度概览                                │
│                                             │
│  ┌─────────┐   成功  40 (88.9%)  ████████  │
│  │  圆环图  │   失败   3 ( 6.7%)  ██        │
│  │  45 次   │   未运行  2 ( 4.4%)  █        │
│  └─────────┘                                │
│                                             │
│  ⚠ 3 个工作流执行失败 → [查看详情]           │
└─────────────────────────────────────────────┘
```

- 左侧 ECharts 圆环图（绿/红/橙）
- 右侧数字摘要 + 进度条
- 底部失败提示行：有失败时显示，点击跳转到调度中心并筛选失败工作流
- 全部成功时底部显示"全部执行成功"绿色文字

**保留：** 快捷操作入口、系统公告，这两个仍然有用。
**考虑去掉：** 平台架构图——对日常使用没有帮助，只在首次了解系统时有用。可以移到「系统监控」页面。

#### 数据来源

后端调用 DS API：
- `GET /projects/{code}/analysis/process-state-count` — 按日期范围查询各状态数量
- `GET /projects/{code}/process-definitions` — 工作流总数
- 近 7 天数据：循环查 7 天的 process-state-count，每天汇总一个数字

后端一次性返回给前端，前端不直接调 DS。

### 5.2 调度中心（Scheduler）— Phase 2

改造量最大的模块。从 iframe 改为两个原生子页面：**工作流管理** + **运行记录**。

#### 5.2.1 工作流管理（/scheduler/workflows）

配置态页面：管理工作流定义、调度上下线、手动触发。

```
┌─────────────────────────────────────────────────┐
│  工作流管理                                       │
│  管理调度工作流定义                                │
│                     [搜索框]  [新建工作流] [去DS后台]│
├─────────────────────────────────────────────────┤
│  [全部 12] [已上线 8] [未上线 4]        ← 状态筛选  │
├─────────────────────────────────────────────────┤
│  工作流名称    调度状态  CRON表达式   最近运行  时间   操作│
│  ─────────────────────────────────────────────── │
│  daily_sync   [已上线]  0 2 * * ?  [失败]  05-10  ▶ ⏸ ⋯│
│  user_etl     [已上线]  0 3 * * ?  [成功]  05-10  ▶ ⏸ ⋯│
│  report_gen   [未上线]  —          [未运行] —      ▶ ▶ ⋯│
│  ...                                             │
├─────────────────────────────────────────────────┤
│  共 12 条                              < 1 2 3 > │
└─────────────────────────────────────────────────┘

操作按钮：▶ 运行  ⏸ 启停调度  ⋯ 更多（编辑、补数、删除）
```

**操作按钮逻辑：**

| 操作 | 条件 | 行为 |
|------|------|------|
| 运行 | 任何时候 | 手动触发一次执行，确认弹窗 |
| 启动调度 | scheduleStatus=OFFLINE | 上线 CRON 调度 |
| 停止调度 | scheduleStatus=ONLINE | 下线 CRON 调度，确认弹窗 |
| 编辑 | 任何时候 | 打开 DS DAG 编辑器模态框 |
| 补数 | 任何时候 | 弹窗选择日期范围，批量执行历史日期 |

**业务合理性：**
- 「运行」= 立即跑一次，「启动调度」= 按 CRON 定时跑，两个概念分开
- 「补数」是数据行业常见需求：某天数据有问题，需要选日期范围重跑

#### 5.2.2 运行记录（/scheduler/history）

运维态页面：查看所有工作流的执行结果，排查问题，重跑失败。**早晨巡检最常用的页面。**

```
┌─────────────────────────────────────────────────┐
│  运行记录                                        │
│  调度执行历史与运维                                │
│                              [日期选择器] [自动刷新]│

│  ┌─ 日历热力图（近30天）──────────────────────┐   │

│                                                   │
│  [全部] [成功 40] [失败 3] [运行中 2]   ← 状态筛选  │
├─────────────────────────────────────────────────┤
│  工作流名称    状态    开始时间       结束时间   耗时  操作│
│  ─────────────────────────────────────────────── │
│  daily_sync   [失败]  05-10 02:00  05-10 02:03  3m  [详情][重跑]│
│  user_etl     [失败]  05-10 03:00  05-10 03:05  5m  [详情][重跑]│
│  report_gen   [成功]  05-10 04:00  05-10 04:12  12m [详情]     │
│  clean_log    [成功]  05-10 05:00  05-10 05:01  1m  [详情]     │
│  ...                                             │
├─────────────────────────────────────────────────┤
│  共 45 条                              < 1 2 3 > │
└─────────────────────────────────────────────────┘
```

**日历热力图（竞品调研新增）：**
- 参考 Airflow Grid View + GitHub 贡献图
- ECharts 日历图，展示近 30 天每日执行健康度
- 颜色：深绿=全部成功、浅绿=大部分成功、红=有失败、灰=无执行
- 点击某天 → 自动筛选该日运行记录
- 一眼看出哪些天有问题，比纯表格直观

**任务详情展开：**

点击「详情」→ 表格行下方展开任务列表（inline 展开，不用抽屉）：

```
│  ▼ daily_sync  [失败]  05-10 02:00  05-10 02:03  3m  [详情][重跑]│
│    ├ ✓ task_extract    [成功] 02:00→02:01  1min                  │
│    ├ ✓ task_transform  [成功] 02:01→02:02  1min                  │
│    └ ✗ task_load       [失败] 02:02→02:03  1min  [日志] [重跑]    │
```

- 展开后显示该次运行的所有任务，缩进呈现层级关系
- 失败任务红色高亮 + 「日志」「重跑」按钮
- 成功任务绿色勾，无需额外操作

**执行耗时趋势（竞品调研新增）：**
- 在详情展开区域底部，一条迷你折线图
- 展示该工作流近 20 次运行的耗时变化
- 耗时突增时一眼看出，提前发现性能劣化

**日志模态框：**
点击「日志」弹出模态框，全屏等宽字体显示任务执行日志：
- 黑底白字的终端风格
- 自动滚到底部（最新日志）
- 日志文本可选中复制
- 关键字高亮：ERROR 红色、WARN 黄色

#### 新建/编辑工作流（模态框内嵌 DS）

```
┌──────────────────────────────────────────┐
│  新建工作流                         [✕]  │
├──────────────────────────────────────────┤
│                                          │
│  ┌──────────────────────────────────┐    │
│  │                                  │    │
│  │    DS DAG 编辑器 (iframe)         │    │
│  │                                  │    │
│  │    拖拽 Shell / SQL / DataX 节点  │    │
│  │    连线定义依赖关系               │    │
│  │                                  │    │
│  └──────────────────────────────────┘    │
│                                          │
└──────────────────────────────────────────┘
```

**设计决策说明：**

为什么不自己做 DAG 编辑器？
- DS 的 DAG 编辑器是其最核心、最复杂的前端组件，涉及拖拽、连线、节点配置、参数传递、依赖校验等
- 自研同等能力需要 2-4 周纯前端开发，且容易有 bug
- 用户创建/编辑工作流是**低频操作**（一周几次），日常高频操作是查看和运维
- 模态框方案让用户 95% 的时间在原生 UI 里，只有 5% 的时间接触 DS 原生界面

**实现细节：**
- 模态框全屏展示（`width: 100vw, height: 100vh`），最大化编辑空间
- iframe 指向 DS 的工作流创建/编辑页面
- 加载前自动设置 sessionId cookie
- 关闭模态框时自动刷新工作流列表
- 模态框标题栏只保留标题和关闭按钮，不加额外操作

### 5.3 系统监控（Monitor）— Phase 4

#### 改造前

6 张服务状态卡片（全部硬编码 online），静态服务器信息，部署架构说明文字。

#### 改造后

**服务状态卡片（保留，改为真实数据）：**

DS 的状态从硬编码改为调用 `/api/ds/monitor` 真实检测。其他服务也尽量做真实健康检查（通过后端 curl 各服务的 health 端点）。

**新增：DolphinScheduler 资源监控区域**

```
┌─────────────────────────────────────────────┐
│  调度引擎资源监控                              │
│                                             │
│  CPU 使用率     ████████░░░░  45%            │
│  内存使用率     ██████████░░  68%   1.3G/2G  │
│  系统负载       1.2                          │
│  最近心跳       2026-05-11 10:05:23          │
│  运行模式       Standalone                   │
│                                             │
└─────────────────────────────────────────────┘
```

- 进度条颜色：<60% 绿色 / 60-80% 黄色 / >80% 红色
- standalone 模式下 master 和 worker 是同一进程，展示一行数据即可
- 内存同时显示百分比和绝对值（1.3G/2G）更直观

**保留但优化：** 服务器信息区域保留，但从硬编码改为尽量从系统获取。
**移入：** Dashboard 的「平台架构图」移到这里更合适（系统级信息归系统监控页面）。

### 5.4 通知系统 — Phase 2 附属

#### 触发机制

DS 工作流/任务失败时，通过 Webhook 回调 Portal 后端：
- DS 侧配置告警组 → HTTP 告警类型 → 回调 `http://portal-backend:8000/api/notifications/ds-webhook`
- Portal 后端接收后存入通知表

#### 前端展示

```
┌───────────────┐
│ 🔔 (红点 3)   │ ← 顶部铃铛，已有，改为真实数据
├───────────────┤
│ 通知           │
│               │
│ 🔴 daily_sync 执行失败     2分钟前  │
│    任务 task_load 出错              │
│    [查看详情]                       │
│                                    │
│ 🔴 user_etl 执行失败       1小时前  │
│    任务 extract 超时                │
│    [查看详情]                       │
│                                    │
│ 🟢 report_gen 调度已上线    3小时前  │
│                                    │
│           [查看全部]                │
└────────────────────────────────────┘
```

- 下拉面板最多显示最近 5 条
- 失败通知红色图标 + 红色文字
- 点击「查看详情」跳转到调度中心对应工作流
- 「查看全部」跳转到通知列表（可以简单做，也可以后续做独立页面）

## 六、DS 功能去留对照表

完整列出 DS 所有菜单项及处理方式：

### DS 首页/仪表盘
- **任务状态统计** → 拆到 Portal 工作台（Phase 1）
- **流程状态统计** → 拆到 Portal 工作台（Phase 1）

### DS 项目管理
- **项目列表/创建/编辑** → 去掉，后端固定一个默认项目

### DS 工作流（项目内）
- **工作流定义列表** → 拆到 Portal 调度中心（Phase 2）
- **工作流定义创建/编辑 (DAG)** → 模态框内嵌保留（Phase 2）
- **工作流实例列表** → 拆到 Portal 调度中心-运行历史（Phase 2）
- **任务实例列表** → 拆到 Portal 调度中心-任务详情（Phase 2）

## 七、技术架构

### 7.1 整体架构变化

```
改造前：
  浏览器 → Portal 前端 → iframe → DS 前端 → DS 后端
                       ↗ (独立的两套认证和 UI)

改造后：
  浏览器 → Portal 前端 → Portal 后端 → DS REST API
                       ↗ (统一认证，统一 UI)
           唯一例外：创建/编辑工作流时模态框内嵌 DS 前端
```

### 7.2 后端代理层设计

Portal FastAPI 后端作为 DS API 的代理和聚合层：

```
Portal 前端
    │
    ▼
Portal FastAPI (/api/ds/*)
    │
    ├── DSClient (单例)
    │   ├── 自动登录 DS，缓存 sessionId
    │   ├── 401 自动重认证
    │   └── 自动发现并缓存默认项目 code
    │
    ├── /api/dashboard/stats      → 聚合：DS 统计 + Portal 统计
    ├── /api/ds/workflows         → 合并：定义 + 调度 + 最近运行
    ├── /api/ds/workflows/{code}/* → 透传：运行/上线/下线/重跑
    ├── /api/ds/instances/{id}/*   → 透传：任务列表/日志
    ├── /api/ds/monitor           → 聚合：Master + Worker 状态
    └── /api/notifications/*      → 独立：通知 CRUD + Webhook 接收
```

**为什么要做代理层而不是前端直接调 DS？**
1. 认证统一：前端只需要 Portal 的 JWT，不需要管 DS 的 sessionId
2. 数据聚合：工作流列表需要合并 3 个 DS API 的数据，后端做更高效
3. 安全隔离：DS 凭据不暴露给前端
4. 接口稳定：即使 DS 升级换 API，只需要改后端代理层

### 7.3 关键实现：DSClient

```python
class DSClient:
    """DS API 客户端单例"""
    
    功能：
    - __init__: 创建 httpx.AsyncClient，从配置读取 DS 地址和凭据
    - _login: POST /login 获取 sessionId
    - _ensure_session: 检查 session 有效性，失效则重新登录
    - _discover_project: GET /projects 获取默认项目 code
    - get/post/put/delete: 便捷方法，自动注入 cookie 和项目 code
    
    错误处理：
    - DS 不可用时返回友好错误，Portal 其他功能不受影响
    - 401 自动重认证，最多重试 1 次
    - 超时设置 30 秒，避免 DS 卡死拖垮 Portal
```

### 8.1 工作量估算

| 阶段 | 后端 | 前端 | 说明 |
|------|------|------|------|

| Phase 1 工作台 | 小 | 中 | 1 个聚合接口 + Dashboard 改造 |
| Phase 4 监控 | 小 | 小 | 1 个接口 + Monitor 加一块区域 |

### 8.2 风险点及对策

| 风险 | 影响 | 对策 |
|------|------|------|
| DS session 过期 | API 调用 401 | DSClient 自动重认证 |
| DS 服务重启 | 短暂不可用 | Portal 其他功能不受影响，DS 相关显示"服务暂不可用" |
| DS 默认项目不存在 | 工作流 API 无法工作 | 启动时自动检测，不存在则自动创建 |
| 模态框内嵌 DS 的 URL 路径 | DS 前端路由可能变化 | 抽取为配置项，方便调整 |
| DS 分析 API 数据量大 | Dashboard 加载慢 | 后端缓存 60 秒，避免每次刷新都调 DS |
| DS Webhook 告警格式 | 可能和预期不一致 | 先人工测试 webhook 格式，再写解析逻辑 |

### 8.3 实施顺序及依赖

```

  └── DSClient + 配置 + httpx 依赖
  └── 验证：curl /api/ds/monitor 返回 DS 状态

第 3 步：Phase 4（监控增强）

  └── Monitor.vue 加 DS 资源区域

第 4 步：Phase 2（调度中心）— 分 4 个子步骤
  └── 2.1 工作流列表 + 基本状态展示
```
